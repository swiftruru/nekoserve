# NekoServe v2.0.0 -- Floor plan rebuild, passive exposure channel, citations story mode, and a 680× playback speed-up

Released: 2026-04-23

v2.0.0 bundles three major pieces of feature work (v2.0 floor plan contract, v2.1 rendering + animations, v2.2 passive-exposure channel), a complete rebuild of the citations page into an interactive story, and a batch of performance + DX fixes that make paper-sized runs feel instant instead of frozen. It is the first release where running the full Hirsch 2025 scenario (35 seats, 27 cats, 720 min) lands on the Playback page without a multi-second main-thread stall.

## v2.1 Floor plan: portrait SVG café with counter-aware paths

The Playback page used to draw the café as abstract colored zone cards. v2 replaces that with a real SVG floor plan that mirrors a physical Stockholm-style cat café layout: portrait canvas, a top "Area 2" lounge with tables and cat shelving, a bar counter that splits the room, a lower "Area 1" with booth seating, a partitioned **Cat Room** with its own food / water bowls and cat flap, and an entrance corridor on the right.

- **Furniture anchors** replace the old "seat grid" — each of the N seats is bound to a specific chair sprite at a specific table, so a 5-seat table behaves like a 5-seat table rather than a generic capacity pool.
- **Counter-aware walking paths** route customers through real aisles instead of slicing through the counter: door → entrance corridor → aisle → chair, with an L-shaped detour around the counter for Area 2 destinations.
- **Cat Room fade** — cats that retreat to the Cat Room smoothly fade their opacity as they cross the partition, so the visual distinction between "interactable" and "resting behind glass" reads at a glance.
- **Shelf hop** — cats use vertical level transitions (floor / furniture / shelf) as real SVG jumps with a quick squash-and-stretch, matching Hirsch 2025's observed vertical-level distribution.

## v2.2 Passive Exposure channel (PE)

Hirsch 2025 only measured the **contact** channel (a cat physically on or next to a customer). In practice cat-café patrons also derive happiness from just *seeing* a cat nearby. v2.2 introduces a second-channel customer-satisfaction model that accumulates "visible but not visiting" cat-minutes per customer without perturbing the existing `customerSatisfactionScore` (which is the baseline our validation mode scores against).

**Rate formula** (hybrid: same-area + Euclidean distance decay + visibility × behavior weights):

```
PE_rate(customer c, time t) = Σ_k  1[same_area(c, k)] · V(k) · D(c, k) · B(k)
```

Integration is piecewise — the rate is constant between event timestamps, so the accumulator only fires at event boundaries. Implementation lives in [`utils/passiveExposure.ts`](src/renderer/src/utils/passiveExposure.ts).

**Results page surface** — a new `PassiveExposureSection` reports three KPIs:

- Average weighted passive-exposure minutes per served customer
- Passive : Active ratio (how much of the cat-time pie is ambient)
- Saturated 0-1 score (`1 − exp(−avg / 15)`) for apples-to-apples comparison with the existing normalized satisfaction metrics

A `PassiveChannelSensitivity` sub-panel lets you slide the vertical-level and distance weights and see the aggregate PE respond live.

## Citations page rebuilt as interactive story

The Citations page used to be a flat list of papers with BibTeX buttons. v2 rebuilds it as a scrollable **story** with nine animated concept modules that illustrate what each cited paper actually taught us. Each module runs in two modes: **ambient** (always-on background animation) and **scripted** (a step-through narration you can click forward/backward).

The nine concept animations, each in its own file under [`components/citations/concepts/`](src/renderer/src/components/citations/concepts/):

- **LittlesLawGauge** — a live `L = λ · W` gauge where drops fall in from above and drain at the bottom
- **BalkReneQueue** — shows the balking / reneging decision points in a queue with avatars turning away
- **ArrivalDropper** — Poisson inter-arrival drops coming from the top at exponentially distributed intervals
- **RenegingFader** — a customer's patience bar fading until they abandon the queue
- **AttributeBars** — the PLS-SEM construct weights from Li et al. 2025 (coffee quality insignificant, pet interactivity dominant)
- **EthogramWheel** — Ropski et al. 2023's observed cat-behavior ethogram as a rotating pie
- **InteractionMatrix** — cat-human interaction percentages from Hirsch 2025's 227-hour dataset
- **VerticalLevelBounce** — Hirsch 2025 Figure 4 vertical-level preferences (floor / furniture / shelf) as stacked bouncing bars
- **WelfareBars** — cat welfare indicators across foster vs. café settings

Supporting components: [`StoryChapter`](src/renderer/src/components/citations/StoryChapter.tsx) and [`StoryNav`](src/renderer/src/components/citations/StoryNav.tsx) wrap the chapters with a sticky side nav; [`CitationCard`](src/renderer/src/components/citations/CitationCard.tsx) is the unified paper card with author / venue / DOI / BibTeX copy; [`CitationLandscapeMap`](src/renderer/src/components/citations/CitationLandscapeMap.tsx) and [`CitationPipelineFlow`](src/renderer/src/components/citations/CitationPipelineFlow.tsx) render the top-level "how these papers connect" diagrams.

All animations use `ScriptedAnim` — a shared base that drives deterministic keyframe playback with pause / scrub / reverse, so the scripted-mode narration is always in sync with the visuals.

## Playback performance: 340× fewer reducer applications

Running the paper-sized scenario used to freeze the main thread for 8-15 seconds between "simulation complete" and the Playback page appearing. Root cause:

```ts
// Old buildSnapshotSeries — O(events × duration)
for (let t = 0; t <= duration; t += step) {
  const state = replayUpTo(ctx, t)  // replays from 0 every iteration
  out.push(snapshotFromState(state, t))
}
```

For the Hirsch scenario (750 minutes × 4617 events), that's **~3.46 million reducer applications** crammed into a single synchronous render pass on both the Playback and Results pages. The fix is a textbook sweep-line pass — carry a running state forward and emit snapshots at step boundaries:

```ts
// New buildSnapshotSeries — O(events + duration)
const state = initialState(ctx.config)
let eventIdx = 0
for (let t = 0; t <= duration; t += step) {
  while (eventIdx < events.length && events[eventIdx].timestamp <= t) {
    applyEvent(state, events[eventIdx], ctx)
    eventIdx += 1
  }
  out.push(snapshotFromState(state, t))
}
```

Total work drops to ~5400 operations — roughly **640× less** for the same paper-sized run. Snapshots only read counters + occupancy aggregates, so the per-frame TTL sweeps that `replayUpTo` does for rendering (bubble expiry, pulse fade, customer linger cleanup, passive exposure flush) are safely skipped in the snapshot pass. The original `replayUpTo` remains unchanged for the actual scene rendering path.

## GlobalRunIndicator: never leave the user staring at a frozen screen

A new always-on-top progress overlay ([`components/GlobalRunIndicator.tsx`](src/renderer/src/components/GlobalRunIndicator.tsx)) lives outside the page tree and stays visible across tab transitions. Three render modes driven by live state:

1. **Batch progress** — real fraction of replications completed (e.g. `執行中（7 / 20）`)
2. **Simulation progress** — real simulated-time fraction streamed from the Python simulator (`模擬時間 5:15 / 12:00`), with a "warmup" badge during the warm-up phase
3. **Rendering** — a synthetic stage we insert after the IPC returns, while React commits the heavy result cascade. Without this the indicator would vanish the moment status flipped to `'success'` and the user would see the long post-result render freeze with no feedback

Fallback: when no live progress has arrived yet (old simulator binary, ultra-fast run), the bar eases exponentially toward 95 % so it's always doing *something*.

The `useSimulation` hook's `run()` function now subscribes to `electronAPI.onSimulationProgress` **before** invoking the IPC call, so the very first progress emit doesn't get lost to a handler-registration race. On completion it keeps `simProgress` non-null across the status flip so the indicator rides through the heavy commit, and the caller invokes `clearSimProgress()` after the destination page has settled (via `requestIdleCallback` with a 5 s timeout fallback).

## Python simulator: first-emit-before-env.run

Old behavior: the `progress_emitter` was registered as a SimPy process that queued behind every other process (27+ cat agents, the customer generator, the warm-up reset), so the initial 0 % emit only fired after the whole SimPy event loop had drained its zero-delay queue. On heavy configs that meant no progress for 3-10 seconds, which the UI can't distinguish from a freeze.

Fix ([`core.py:429-455`](simulator-python/simulator/core.py#L429-L455)): hoist the 0 % emit out of the SimPy process and write it directly to `stdout` before `env.run()` is called. The periodic 5-min emits stay inside the SimPy process. The UI now snaps to 0 % the instant Python starts executing, and the bar updates smoothly as the simulation progresses.

## Validation warnings: typed i18n keys

`ValidationReport.warnings` used to be `string[]` of hardcoded English sentences, which meant the validation page couldn't localize them. v2 makes warnings a `ValidationWarning[]` with:

```ts
interface ValidationWarning {
  key: string                              // i18n key under `validation` namespace
  params: Record<string, string | number>  // interpolation values
}
```

Four warning variants replace the two old stringly-typed ones: `warn.abandonRate.tooHigh` / `warn.abandonRate.tooLow` / `warn.noCatVisitRate.tooHigh` / `warn.noCatVisitRate.tooLow`, each with observed / min / max interpolation. The Validation page now renders them through `useTranslation`, so a zh-TW user sees "放棄率 18.5% 高於文獻範圍 [5%, 15%]" instead of an English sentence glued into the report.

## Developer experience fixes

- **DevTools no longer auto-pop on `npm run dev`** ([`main/index.ts:312-321`](src/main/index.ts#L312-L321)) — it was noisy every reload. Opt in with `NEKOSERVE_DEVTOOLS=1 npm run dev` or press `Cmd+Option+I` (the application menu wires `toggleDevTools` so the shortcut still works after the View menu was stripped).
- **Dev mode always tracks the Python source** ([`simulator-bridge.ts:110-122`](src/main/simulator-bridge.ts#L110-L122)) — the old fallback silently preferred a stale PyInstaller binary under `dist/simulator/` when it existed, which masked Python edits during development. Dev now unconditionally runs `python -u -m simulator`; the binary path is only resolved in packaged builds. Packaged builds are unaffected.
- **Simulator timeout 30 s → 5 min** ([`simulator-bridge.ts:18`](src/main/simulator-bridge.ts#L18)) — paper-sized runs (27 cats × 750 min) legitimately need 30-60 s of wall time on typical hardware, and the 30-second SIGKILL was firing on the Hirsch scenario before it could finish. Five minutes is a comfortable ceiling; if you hit it, something is genuinely wrong.

## Author's note

v2 is the release where NekoServe stops *describing* the Stockholm cat café and starts *looking* like one. The floor plan, the dual-channel satisfaction model, the scripted citations story, and the paper-faithful scenario preset all pull toward the same thing: someone opening the app for the first time should be able to click "論文樣本" and immediately see the Hirsch 2025 paper play out on screen, with the validator agreeing. The performance fixes were the final blocker — no amount of beautiful rendering saves an app that freezes for 15 seconds on the canonical demo config.
