# NekoServe v2.4.0 -- Live Convergence Mode: watch the answer settle as the batch runs

Released: 2026-05-08

v2.4.0 adds a brand-new **Live Mode** page (`⚡` in the nav) that turns batch replication from "press run, wait, look at one CI" into a streaming, classroom-friendly experience: you launch N runs and immediately watch the cumulative-mean curve climb, the 95% CI band tighten as `1/√n`, and the histogram morph from a sparse single bar into a bell. The page also auto-detects when the answer is stable and tells you "you can stop now". This is the version that finally makes the *"how many runs is enough?"* question something a student can see, not just trust.

## The new Live Mode page

**Streaming runner.** A new `useLiveRunner` hook drives one replication at a time and pumps every completed run into a Zustand store (`liveBatchStore`) so the page can re-render incrementally. The runner exposes pause / resume / stop, surfaces per-run errors without aborting the batch, and hands a partial-results array to the commit step when the user stops early -- so a 50-run batch the user halts at run 27 still lands a valid 27-run summary on the Results page.

**`CumulativeMeanChart`.** Orange cumulative-mean line with a light-orange 95% CI band that visibly contracts as `1/√n`. A pulsing dot marks the most recent run, a gray dashed reference line shows run 1's lone value (so the gap between "trust one run" and "trust the average" is literally drawn), and a green stable zone auto-shades the segment where the cumulative mean changed by less than 1% over the trailing 100 runs. Click any point to inspect that run's exact mean and CI.

**`LiveHistogram`.** 20-bin histogram of the chosen metric with an overlaid KDE smoother (`utils/kde.ts`), median + P5 + P95 reference lines, and an auto-classified shape verdict in the corner ("approximately normal", "right-skewed (long tail right)", "long-tail distribution (heavy outliers)" -- driven by `utils/distributionShape.ts`). The bell-shape emerges live as samples accumulate.

**`MetricSmallMultiples` + `MetricSelectionPanel`.** A grid of mini cumulative-mean charts, one per selected metric (default: 6, configurable, persisted). Click any tile to drill into the full `CumulativeMeanChart` + `LiveHistogram` view for that metric.

**`LiveScenePanel` + `RunThumbnailStrip`.** While the batch runs, the left half of the page plays the **same SVG café animation** as the regular Playback page, but cycles through every completed run as it lands -- so the cat scene visibly changes shape between runs even though the config is fixed. A horizontal strip of completed-run chips lets the user click any past run to scrub the animation back to that one. The cycling is gated on playback being paused / ended so we don't yank the user mid-replay; the swap effect uses a `pendingSwapRef` + `pendingSwapTick` pattern (App.tsx) to re-evaluate without unnecessary re-renders.

**Convergence detector + celebration.** `utils/convergence.ts` watches the trailing 100 cumulative means and flips to "stable" when the relative change drops under 1%. When that happens the chart shades a green zone, the convergence hint tells the user "✓ Stable: you can stop here", and a one-shot `ConfettiBurst` plays. The threshold and window are surfaced to the user in the hint copy, so it's a teachable signal, not a magic light.

**Replay curves.** A "Replay curves" button redraws every cumulative-mean / histogram bin from `n=1` to the end, animating the growth -- useful for showing the convergence story to an audience after the batch is done.

**Empty state.** If a user lands on Live Mode without ever having run a batch, the page explains what the mode is for ("answers the *how many runs is enough?* question"), why it needs `≥ 2 runs`, and offers two paths back: navigate to Settings and toggle batch mode on, or **🚀 Run a 20-replica batch with the current params** in one click.

**Glossary additions.** Two new tooltipped terms (`bigN` / `lowerN`) so the chart copy can use the statistical convention -- **N** = total planned, **n** = completed so far -- without confusing first-time viewers. Both pages and every chart caption explain the distinction inline.

## Discoverability nudges from the rest of the app

**Settings batchHint card.** Above the batch / sweep mode toggles, when both are off, a dashed-orange card explains: *"A single run gives you ONE answer. A batch tells you HOW STABLE that answer is."* It compares "1 run → one dot (no idea if accurate)" vs "20 runs → a curve + CI band (see stability)" so users know what to expect from the new page before they enable batch mode. Hidden once batch mode is on -- once they get it, the hint becomes noise.

**Playback BatchPromoBanner.** After a single-run playback, a one-time orange banner pops up on the Playback page nudging *"Want to know if this result is stable?"* with a one-click **"🚀 Run 20-replica batch now"** button that reuses the current config. Hidden permanently after the user runs their first batch (`liveBatchStore.runs.length > 0`) or clicks the explicit dismiss link (persisted in `localStorage`).

## Routing changes

The batch runner used to land users on the Playback page after the batch finished. v2.4 routes batch runs to the new Live Mode page **at the start** of the batch, hydrates `result` with the first completed run so the embedded `LiveScenePanel` can start animating immediately, and only commits a `BatchResult` to history once all runs (or the runs the user kept after Stop) are in. The old `runBatch` flow is replaced by `beginLiveBatch` + `commitLiveBatch` on `useSimulation`; `setLiveFirstRun` lets App hand a fresh result to the embedded scene mid-stream without waiting for the commit. PlaybackPage gets a new `onPlayingChange` callback so App knows when it can safely swap the scene to the next-finished run.

## E2E mock jitter

The screenshot-capture suite needs realistic-looking curves and histograms, but the mock simulator returns the same fixture every call -- which would make every cumulative-mean curve flat and every histogram a single bar. Two new env vars now control opt-in Gaussian jitter on the mock fixture's metrics:

- `NEKOSERVE_E2E_SIMULATION_JITTER=1` -- enable jitter
- `NEKOSERVE_E2E_SIMULATION_JITTER_SEED=<int>` -- seed the LCG so successive screenshot test runs produce comparable distributions

A per-process LCG drives Box-Muller Gaussian draws with metric-aware coefficient-of-variation (rates / utilisations: 8%; P95/P99 latency: 12%; everything else: 10%), then clamps to each metric's natural range (`[0,1]` for rates / utilisations / satisfaction, `[0,5]` for `catWelfareScore`, `[0,∞)` for waits / times / counts). Jitter is off by default and gated behind the env var, so production runs are byte-identical to before. Both env keys are added to `tests/ui/helpers/electronApp.ts` `PASSTHROUGH_ENV_KEYS` so Playwright tests can pass them through.

The matching `tests/screenshots/capture-v4.spec.ts` orchestrates the v2.4 weekly-report capture: jitter on, batch run launched, screenshots taken at multiple n checkpoints (run 1 alone, run 5, run 20, run 50) so the convergence story is visible in stills.

## i18n

A new `liveOverlay` namespace lands in both locales with full bilingual coverage:

- chart axis / element / interpretation help text (`chart.help.*`, `histogram.help.*`)
- the N vs n notation explainer (`notationN` / `notationLowerN` / `notationWhyDifferent`)
- distribution-shape verdicts (`shape.normal`, `shape.rightSkewed`, `shape.longTail`, ...)
- empty-state copy with the "go to settings" + "quick-start 20-replica batch" CTAs
- 21 metric labels for the metric selector (`metric.avgWaitForSeat` ... `metric.customerSatisfactionScore`)

`nav.json` gains the `liveMode` entry; `playback.json` gains the `batchPromo.*` block; `settings.json` gains the `batchHint.*` block plus `batch.quickStartLabel`; `results.json` gains the `bigN` / `lowerN` glossary entries.
