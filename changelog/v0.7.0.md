# NekoServe v0.7.0 — Update checking

Released: 2026-04-16

v0.7.0 adds built-in update checking against GitHub Releases. The app can now detect when a newer version is available and guide the user to the download page. No external dependencies added; the feature uses Electron's built-in `net.fetch` and a simple semver comparison. No simulator changes, no event schema changes, no breaking changes.

## Update checking

### Automatic check on launch

Five seconds after startup, the app silently queries the GitHub Releases API (`/repos/swiftruru/nekoserve/releases/latest`). If a newer version exists and the user hasn't previously skipped it, a modal notification appears. If the app is already up-to-date or the network is unavailable, nothing happens.

### Manual check via menu

- **macOS**: *NekoServe > Check for Updates...*  (in the app menu, right below "About NekoServe")
- **Windows / Linux**: *Help > Check for Updates...*

Mirrors the pattern used by VS Code and other Electron apps.

### Three-action update modal

When a new version is found, the user sees a centered modal with:

1. **Go to Download** (primary orange button) — opens the GitHub Releases page in the system browser.
2. **Skip This Version** (secondary outlined button) — persists the choice to `userData/update-prefs.json`. The app will not prompt for this specific version again, even on relaunch.
3. **Remind Me Later** (tertiary text link) — dismisses the modal for this session. The next app launch will check and prompt again (5-second rule).

Additional states:
- **Checking**: spinner with "Checking for updates..." text.
- **Up to date** (manual check only): app icon + confirmation message + OK button.
- **Error** (manual check only): warning icon + error detail + OK button.

### About page: Version & Updates card

A new card at the bottom of the About page shows the app icon, current version, and a "Check for Updates" button. Clicking the button triggers the same manual check flow as the menu item.

The About page version display was also fixed: it now reads dynamically from `app.getVersion()` instead of a hardcoded i18n key that had fallen out of sync (previously showed v0.6.0 when the actual version was 0.6.1).

## Architecture

Four new modules under `src/main/updater/`:

| Module | Responsibility |
|--------|---------------|
| `update-config.ts` | GitHub owner/repo, API URL, Releases URL, timing constants |
| `update-service.ts` | `checkForUpdate()`: fetches GitHub API, parses response, compares semver |
| `update-store.ts` | Read/write skipped-version preference to a JSON file in userData |
| `update-ipc.ts` | Registers IPC handlers (`check-for-update`, `skip-update-version`, `open-releases-page`) and orchestrates the 5-second auto-check |

Renderer side:

| File | Responsibility |
|------|---------------|
| `useUpdateCheck.ts` | React hook managing update state, listening for main-process push events and menu triggers |
| `UpdateModal.tsx` | Modal component with four visual states (checking, available, up-to-date, error) |

Preload exposes five new APIs on `window.electronAPI`: `checkForUpdate`, `skipUpdateVersion`, `openReleasesPage`, `onUpdateAvailable`, `onMenuCheckForUpdate`.

## Platform strategy

All current build targets (macOS DMG/ZIP, Windows portable EXE, Linux AppImage) use the "check + redirect to GitHub Releases" flow. None of these formats support in-place auto-update.

The module structure is designed so that a future Windows NSIS installer target could add auto-download-and-install (via `electron-updater`) without changing the existing redirect-based code paths.

## Other changes

- **Fade-in animation**: a new `fade-in` keyframe in `tailwind.config.js` (0.15s ease-out scale + opacity) used by the update modal.
- **App icon in renderer assets**: `src/renderer/src/assets/app-icon.png` is now available for import in renderer components (used by UpdateModal and AboutPage).
- **Main-process i18n**: two new menu strings (`checkForUpdates`, `help`) in both zh-TW and en.
- **Renderer i18n**: new `update` namespace with 15 keys in both zh-TW and en.

## Upgrade notes

- Version bumped `0.6.1` -> `0.7.0`.
- **No breaking changes**. No simulator changes, no event schema changes, no metric field changes.
- Five new IPC channels added to the preload bridge. Existing channels unchanged.
- One new file in userData (`update-prefs.json`) created on first "Skip This Version" action.
